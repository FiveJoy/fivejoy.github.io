1、My Usage

将eclipse的project commit到GitHub

2、Git特点

  • 在本地建立资源库 【这将称作“分布式的版本控制”】
  • 使用git init xxx_directory时xxx_directory中将出现.git，
    自init以后的所有commit对项目的改动都保存在.git中进行本地持久化
  • 无论是将代码分享给github.com还是其他电脑【不同的远端remote】都可将具体的address写在配置文件中–用于发送或者获取资源Push & Pull

    在git上存储图片资源过程
    mkdir pic
    cd pic
    git init pic //初始化GitHub仓库
    git add git01.png
    git commit -m “create pic directory”
    git remote origin add xxx.git //链接到远程库
    git push origin master //推送到远程库
    在GitHub中点击图片用新标签页打开即可得到图片资源链接

3、配置git

  • 需要git命令行工具&Git图形工具&GitHub账户
  • 使用git bash写命令

    git config –global user.name XXX
    git config –global user.email xxx@xx.com

  • 配置行尾 不同平台CRLF&LF比较烦人

    git config –global core.autocrlf true
    git config –global core.autocrlf input

  • 设置颜色

    git config –global color.ui auto

  • local > global > system 设置级别

4、常用命令

init

  • 项目在本地已经存在-想把它做版本控制

    cd Directory
    git init //进入该项目目录直接git init即可【将Directory变成Git仓库】

  • 目录不存在 直接git init Directory即可

git add

git add xxx.txt xxxx.html ——将文本文件存到暂存区的持久化容器中

commit

git commit -m “描述信息” 只有add到暂存区的文件才会被commit到branch中

暂存区的存在可以让你决定哪些修改了的文件不想提交可以不提交

git status——用于帮助查看在上次提交到现在哪些文件做了改动或者增删

  • 查看git仓库状态

git diff

  • git diff 会得到 还未提交到暂存区的文件做了哪些改动—即只针对目前的工作区
  • git diff –staged 暂存区的文件与最近提交的历史文件之间的区别
  • git diff HEAD 工作区文件与最近提交的历史文件之间的区别（和暂存区文件状态无关）
  • 比较两次提交 ？
  • 比较两个branch ？
  • git diff –color-words / git diff –word-diff 查看很长的一行中出现的小改动（直接使用git diff 会显示整个一行）
  • git diff -stat 表示只展示被修改部分的代码块

log

  • 进入git仓库直接使用git log 能看到从最开始到最近一次的跳情况

    提交引用
    提交者的用户名和邮件地址
    提交时间
    commit message

  • git log –oneline 简短的log

    commit index
    commit message

  • git log –stat 更加复杂的log

    commit index
    commit message
    提交包含的文件（甚至包括其路径）
    提交的文件所发生的相对改变【通过 + - 表示】

  • git log –patch 两次提交之间内容的改变
    -可以组合各个选项

  • git log –graph –all –decorate –oneline

remove

  • git rm xxx.file 表示从file system中删除文件并将删除信息存到暂存区中。
  • rm xxx.file ; git rm xxx.file 和上一个的效果是一样的
  • git add -u . 如果多次使用 rm xxx 删除了多个文件，可以使用 git add -u . 遍历当前目录找出已经在file system中删除但还没讲删除信息存到暂存区的文件，并仍然加入暂存区
  • git rm –cached xxx.file 表示不从file system中删除，但是将删除信息保存到暂存区->意味着不再追踪该文件了

git mv 在git中重命名&移动文件是一件事情

  • git mv xxx.jpg src/xx1.jpg

    ① 原位置的xxx.jpg消失
    ② 新出现了src/xx1.jpg

  • 等同于

    mv xxx.jpg src/xx1.jpg
    git rm xxx.jpg
    git add /xx1.jpg

  • git add -A . 从当前目录开始无限向下递归找出被mv的文件并加入Git的暂存区

  • git log –stat – 某个文件的路径 可以展示该文件的历史提交【此时如果该文件存在移动-则无法log】
  • git log –stat -M –follow – 该文件的路径可以告诉日志显示文件移动过程中跟踪文件

    注意在git mv中有一个相似度的阈值（可以用-M来修改）具体怎么回事并不太明白
    特修斯之船一块木板坏了换一块-当所有的木板都被换了一遍时候还是特修斯之船了么？ 50%改变的时候就不是了–50%即是阈值

git push origin(目的地地址) 将全部内容推送到远端

reset 三种模式

1 . soft 选取一条或多条commit，将其全部改变放到暂存区，然后在此基础上创建新的提交 git reset –soft HEAD~5将最新的5条commit都放回暂存区，然后将5条commit压缩为一个，然后直接提交

  • 当意识到多个commit都是涉及的一个事件
    2 . mixed（默认的） 改变历史、改变工作目录（history&working directory）
    git reset HEAD 将暂存区的改变移除（移动回工作区-工作目录working directory）
    3 . hard 即擦除再也不想要的操作
    git reset –hard HEAD~3将最新的3条commit删除【应该是对文本的修改等也是永久的删除了】
    4 . reset和checkout的区别
    reset针对的是整个仓库的改变历史【撤销或者改变某次提交】
    checkout更加精细-针对目录或者文件级别【回到特定文件的某条提交历史COMMIT_INDEX上】
    git checkout file COMMIT_INDEX 将该INDEX下的该文件拉到当前工作区内

reflog
记录checkout commit rebase reset 通过7位的Hash码记录历史
git reset --hard xxxxx(7位hash码)

rebase
git rebase可以将已经存在的提交 放到今天开始的一个分支上
即任何时候都可以选择创建一个新的分支，然后把想要包含的变更的提交包含进去–使其比如说 专注于一个特性

git checkout xxx_branch
git rebase master 意为将当前xxx_branch的工作内容合并到master分支中
和merge不同的是，rebase是将当前分支的提交复写到master分支中
最后形成的效果就是-该Git仓库的历史变化路径是一条直线
因此所形成了良好的阅读体验–>使用rebase而不是merge【历史路径清晰】

对于持续发布的项目（web services , web app等【需求一个更快的发布机制】）最好使用merge–如果一个分支的改变没有达到需求，可以创建一个新的分支来书写满足需求，然后把之前的改动合并掉【快速发表特色分支】
